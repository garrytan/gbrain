# Search Mode Evaluation Methodology

This document explains how Cortex compares retrieval modes such as
`conservative`, `balanced`, and `tokenmax`. The goal is to make mode changes
auditable for tenant operators, CI reviewers, and investor-demo runbooks.

## Scope

This methodology measures:

- retrieval quality on fixed benchmark corpora
- recall and ranking movement across search modes
- estimated downstream model cost from retrieved context size
- latency movement in deterministic replay

It does not measure:

- a specific tenant's live query distribution
- end-user satisfaction
- concurrent production load
- billing for the full agent loop outside retrieved context

For tenant-specific recommendations, run:

```bash
cortex search stats --days 30
cortex search tune
```

after a representative usage window.

## Datasets

| Dataset | Purpose |
|---|---|
| LongMemEval | Long-context QA and retrieval-depth checks |
| Replay captures | Tenant-safe NDJSON captures from synthetic or tenant-approved staging data |
| Cortex fixture qrels | Fixed documents plus binary relevance judgments |

Reported benchmark results must pin dataset identifiers, sample size, seed, and
the Cortex commit SHA used for the run.

## Sample Selection

- Seeded runs use `--seed 42` unless the report says otherwise.
- Questions are not curated per mode.
- The same dataset, seed, and limits feed every compared mode.
- Optional LLM expansion is reported as a separate variable because it can be
  non-deterministic.

## Reproduction Command

```bash
# Setup: run in the Cortex working tree with required provider keys exported.
git rev-parse HEAD

cortex eval run-all \
  --modes conservative,balanced,tokenmax \
  --suites longmemeval,replay \
  --seed 42 \
  --limit 500 \
  --budget-usd-retrieval 5 \
  --budget-usd-answer 20 \
  --output docs/eval/results/current/

cortex eval compare --md > docs/eval/results/current/README.md
cortex eval compare --json > docs/eval/results/current/comparison.json
```

Every run record should include:

- run id
- timestamp
- suite
- mode
- commit SHA
- seed
- limits and parameters
- status
- duration

## Threats To Validity

| Risk | Why It Matters |
|---|---|
| Benchmark domain skew | A dataset heavy in technical content may not predict legal, sales, support, or design-heavy tenants |
| Small fixture size | Directional deltas are more reliable than absolute scores |
| Token heuristic | Cost estimates use approximate token counts unless model-native tokenizers are available |
| Query expansion variance | LLM-generated expansions can vary by provider, model, and prompt revision |
| Bootstrap assumptions | Question-level independence can be imperfect for multi-hop conversations |
| Cache behavior | In-memory benchmark tenants do not model long-running production cache warmth |

## Pre-Registered Expectations

Before publishing a mode comparison, write the expectations in the report. A
typical release might expect:

1. `tokenmax` wins Recall@10 for rare-entity and multi-hop queries.
2. `conservative` wins cost-per-query.
3. `balanced` stays close to `tokenmax` on recall for common queries.
4. No mode drops below the release threshold for nDCG@10.

If the data disagrees, document that directly. Pre-registration makes the
comparison harder to tune after the fact.

## Statistical Discipline

When `cortex eval compare --md` reports a delta between modes, it should include:

- paired bootstrap confidence intervals
- Bonferroni-corrected p-values for multi-comparison reports
- a plain statement when a difference is not statistically significant

Metrics must be defined in `docs/eval/METRIC_GLOSSARY.md` and regenerated by
the glossary freshness check when definitions change.

## Cost Anchors

Search mode cost is driven by delivered chunks:

| Mode | Typical Context Shape | Best Fit |
|---|---|---|
| `conservative` | Tight context, low chunk count | Cost-sensitive agents and quick lookups |
| `balanced` | Moderate context depth | Default tenant workflows |
| `tokenmax` | Large context depth | Frontier model synthesis and multi-hop analysis |

The downstream model tier is usually the larger cost lever because the model
prices the full prompt, not only retrieved chunks. Match mode to model capacity:

- small and fast models pair best with `conservative`
- general reasoning models pair best with `balanced`
- frontier models with large context windows can justify `tokenmax`

## Re-Run Cadence

Regenerate mode comparison reports for releases that touch:

- `src/core/search/**`
- `src/core/embedding.ts`
- `src/core/chunkers/**`
- provider recipe files
- query/search operation handlers
- connector normalization that changes slugs, chunks, or source metadata

Docs-only and UI-only changes do not need the full mode comparison unless they
change query behavior.

## Reproducibility Footer

Every published comparison should end with:

- Cortex commit SHA
- dataset identifiers
- seed
- run commands
- API model identifiers
- pricing snapshot date

Without this footer, the numbers are not falsifiable. With it, another operator
can rerun or rescore the comparison.
