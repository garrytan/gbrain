## 3. Schema / migration changes

### 3.1 Schema deltas

Both embedded/Postgres schema paths changed (`src/core/schema-embedded.ts`, `src/core/pglite-schema.ts`, also reflected through `src/schema.sql`). Key deltas:

1. Links are now multi-type-safe
- old uniqueness: `UNIQUE(from_page_id, to_page_id)`
- new uniqueness: `UNIQUE(from_page_id, to_page_id, link_type)`
- implication: multiple typed edges can now coexist between the same page pair.

2. Timeline entries gain dedup semantics
- new unique index: `idx_timeline_dedup ON timeline_entries(page_id, date, summary)`
- implication: extract/backfill becomes idempotent at the DB layer.

3. Timeline search trigger removed
- dropped `trg_timeline_search_vector` and helper function
- implication: structured `timeline_entries` no longer re-touch page search vectors; markdown timeline content remains the searchable source.

4. Minions job queue tables added
- `minion_jobs`
- `minion_inbox`
- `minion_attachments`
- with claim/stall/delay/timeout/idempotency indexes
- Postgres path additionally installs `pg_notify` trigger on minion job state changes

5. Embedded schema enables `pgcrypto` fallback
- `CREATE EXTENSION IF NOT EXISTS pgcrypto`
- implication: safer UUID support across older Postgres installs.

### 3.2 Migration registry growth

`src/core/migrate.ts` grows baseline migrations 5-10:
- v5 `minion_jobs_table`
- v6 `agent_orchestration_primitives`
- v7 `agent_parity_layer`
- v8 `multi_type_links_constraint`
- v9 `timeline_dedup_index`
- v10 `drop_timeline_search_trigger`

### 3.3 New migration orchestrators

New command-layer orchestrators:

## 4. Breaking changes / absorption risk for our fork

These are the upstream changes we cannot ignore if we rebase onto v0.12.3.

### High-risk contract breaks

1. BrainEngine interface break
- Any local engine wrapper or monkeypatch in our fork must implement the new methods/signatures listed above.
- Most likely breakpoints: compile errors in engine classes, test failures around `removeLink`, `addTimelineEntry`, or any code assuming node-only graph traversal.

2. Operations/MCP surface changed materially
- Because CLI and MCP are generated from `operations.ts`, our Action Brain integrations must not assume the old operation set is complete.
- If we patch or wrap `operations.ts`, we need a careful merge, not a blind overwrite.

3. `put_page` semantics changed
- `put_page` now optionally auto-links after import and exposes `auto_links` in the result.
- Remote callers (`ctx.remote === true`) intentionally skip auto-link for security reasons.
- Any Action Brain workflow relying on pre-v0.12.0 `put_page` behavior needs review, especially if it expected page writes to be graph-neutral.

4. File upload trust boundary hardened
- `OperationContext` now carries `remote?: boolean`.
- `file_upload`-style path validation distinguishes local CLI (`remote=false`) from remote/MCP callers (strict cwd confinement, symlink rejection).
- Any fork code that omitted `remote` or assumed local-style filesystem access from remote callers can break or become intentionally blocked.

### Data / migration absorption risk

5. Postgres brains require JSONB repair awareness
- v0.12.2 introduced a repair path because older Postgres-backed brains stored JSONB as string literals.
- If our fork has Postgres data created on older bases, rebase planning must include `repair-jsonb` / migration orchestration, not just code merge.

6. Schema expectations changed for links and timeline
- multi-type link uniqueness changes query semantics
- timeline dedup index changes insert behavior
- removing the timeline search trigger changes some search/update behavior
- any Action Brain logic that depended on old duplicate/touch behavior needs explicit review

7. Minions schema and jobs API are now first-class
- upstream added queue tables, commands, operations, tests, docs, and orchestration conventions
- if our fork ignores this subsystem, we still need to merge around it cleanly because it touches schema, CLI, operations, and docs

### Process / repo-shape break risk

8. New skillpack / agent-host assumptions
- upstream now ships `skills/**`, migration skills, AGENTS templates, and host-rewrite logic
- if our fork has custom agent conventions, the merge must decide where upstream conventions win vs where Action Brain conventions stay fork-specific

9. CLI help and docs changed enough that downstream docs drift is guaranteed
- `README.md`, `CHANGELOG.md`, `CLAUDE.md`, `docs/GBRAIN_VERIFY.md`, `docs/GBRAIN_SKILLPACK.md` all changed materially
- if we keep our current docs untouched during rebase, user-facing instructions will lie

## 7. Bottom line

This upstream jump is not just “some new commands.” It is a platform step:
- graph auto-wiring became real
- minions/job orchestration became a first-class subsystem
- repair/migration tooling became mandatory operational surface
- engine and operation contracts widened

The merge is feasible because it is mostly additive and has zero renames, but it is not trivial because the additive changes hit exactly the shared seams our fork is most likely to touch: CLI routing, operation contracts, engine interfaces, schema, and agent conventions.
