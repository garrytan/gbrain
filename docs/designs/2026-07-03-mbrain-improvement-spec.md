- **§1** states the goal and the four things wrong today.
- **§2** is the roadmap table — the single source of truth for what ships when.

Later waves detail the high-value items and compress the rest into one-liners with inline evidence anchors; their original per-item acceptance tests live in the pre-rewrite spec (this file's git history). A **[GATE]** tag means the item must not flip a default until the measurement harness (Wave 2) shows no regression. A **[FLAG]** tag means it ships behind a config flag, default off, with a deterministic fallback (offline-parity requirement).

## 1. Goal and diagnosis

### 1.1 What mbrain is for

mbrain is a local-first personal memory runtime for one person and their AI agents. Markdown is the human-editable source of truth; Postgres (or SQLite offline) is the index and operational substrate; CLI and MCP are generated from one contract (`src/core/operations.ts`). The product goal is **compounding context — learn once, never rediscover.** Success is measured at one moment: when an agent needs a fact to do work, does mbrain return the right evidence?

Everything in this spec serves that moment. The review found four ways the system fails it, plus one meta-failure that makes the others hard to fix safely.

### 1.2 The four failures

**① The governance loop leaks, silently.** The write ladder (route → candidate → verify → promote → page) is individually rigorous but does not *close*. Five verified leaks, all silent:
- auto-promote advances an unverified candidate to a dead-end status, then can never see it again (GV-1);
- a promoted candidate whose page was never written counts as "done" (GV-2);
- the "daily memory report" the agent rules call the *primary review surface* is generated by no scheduled process — if the owner never runs it manually, nothing ever alarms (GV-3);
- the highest-authority evidence (a direct user statement) can evaporate when a 15-minute write session expires with no fallback (GV-4);
- one hung phase wedges the nightly maintenance cycle forever — phases have no timeout, and the sweep that marks the job dead cannot cancel the still-running work (GV-5).

The pattern: mbrain is excellent at *not writing wrongly* and blind to *not having written*. Debt accumulates where no one is looking.

**② Korean retrieval is broken (empirically verified on both engines).** The owner works in Korean. On the SQLite/offline profile, a pure-Korean query returns **zero** keyword results — `extractSearchTerms` splits on `[^A-Za-z0-9]` and deletes every Hangul character before it reaches the (Unicode-capable) FTS index. On Postgres, `to_tsvector('english')` matches only exact eojeol, so `레벨리온` misses a document containing `레벨리온은`. The multilingual vector leg is the only path that understands Korean — and it is default-off. This is a correctness bug, not a tuning question.

**③ Knowledge accumulates but does not compound.** The product metaphor is *compiled truth*: a page's top zone is "the current best understanding," rewritten as evidence arrives. But nothing rewrites it. Of the 15 dream-cycle phases, 11 only count things; the compiler that would distill 20 new timeline entries back into a cleaner compiled zone does not exist — that work lives in skill prose an agent may or may not run. Pages have no dedup at write time (a page exists verbatim under two slugs today), no supersession, no usage signal, no per-page health. The brain gets bigger, not wiser.

**④ Retrieval is unwired and context-blind.** Roughly a quarter of the ~180 operations build orientation structures (context maps, atlases, graph frontiers, route planners) that the retrieval path never consults. Manifest aliases and tags — the brain's entity vocabulary — feed no search channel. The graph-frontier planner is fully built and never fed. `retrieve_context` classifies the query's route and then throws the classification away (`route: null` always). And chunks are embedded as raw body slices with no page identity: "he decided to postpone it" enters vector space attached to no entity, no title, no date.

**⑤ (Meta) There is no instrument.** None of ③/④ can be fixed safely by feel, because mbrain cannot currently measure whether a retrieval change helped or hurt. The only live-index recall test is 8 synthetic cases against a fake embedder; the quality "gate" scores injected candidate lists, never live search. This is why measurement is its own wave (Wave 2) and gates every ranking change (Wave 3). It is also, per the Memora comparison below, mbrain's single largest real gap.

### 1.3 What Memora contributes (and what it doesn't)

The review included a code-grounded comparison against Microsoft Research's **Memora** (arXiv 2602.03315; local checkout at `reference/Memora`). Memora's headline idea is a *harmonic memory representation*: never embed the full text, embed a short generated "abstraction" handle plus a few `[Entity]+[Aspect]` cue anchors that dereference to it.

The honest finding: **mbrain's retrieval is already more mature than Memora's on shared mechanics** (dual-engine BM25 + vector + RRF, four-layer dedup, source ranking, a governed evidence boundary). Only three Memora ideas survive as work here, and all are gated:
- its **benchmark harness** → EV-1 (adopt the metric definitions, not the Python);
- its **deterministic trajectory scorecard** `J = G − R − C` → EV-2;
- its **abstraction/cue-anchor index lanes and relevance filter** → AL-1 / AL-2 / RF-1, *deferred* behind the cheaper deterministic changes (RW-1, RS-2, RS-6) that capture most of the same benefit at near-zero cost.

Memora's in-engine LLM retrieval loop, delete-and-rewrite JSON history, dangling-pointer graphs, and GRPO/RL stack are **not ported** — mbrain's caller is already a reasoning agent, and mbrain has referential integrity and a governed ladder Memora lacks (§9).

### 1.4 Constraints every item obeys

1. **Offline parity.** An install with no API key must behave the same at the contract boundary, degrading *honestly* (a visible "unavailable" — never a silent wrong answer). Anything needing an LLM or embedder is `[FLAG]` default-off with a deterministic fallback. Pattern to copy: `governedProbeHybridEnabled` (`search/governed-probe.ts:22`).
2. **The evidence boundary is sacred.** `read_context` returns *evidence*; everything else (candidates, orientation, anchors, frontier nodes) is a *pointer*. New rows in `content_chunks` (e.g. AL-2 anchors) must be excluded from every `read_context` selector.
3. **Additive, never destructive.** mbrain reads body text, so it cannot drop body embedding the way Memora does. Every index idea is an added lane, not a swap.
4. **Referential integrity over string graphs.** Use `page_id` FKs and the `links` table; never `||`-delimited pointers.
5. **Measure before flipping.** No ranking/fusion default changes without a Wave-2 harness artifact (constraint ⑤).

## 3. Wave 0 — Close the loop

### 3.1 Governance

#### GV-1 · Verify before advancing in the auto-promote gate
- **Problem.** A candidate that draws a "promote" verdict but is unverified gets advanced to `staged_for_review`, then the promotion itself throws — and the advance is never rolled back. Because auto-promote only ever scans `captured` and `candidate` status, that candidate is now permanently invisible to every future run. Net effect: silent one-way loss, plus a wasted LLM judge call each time.
- **Evidence.** `promote-gate.ts:120-123` calls `advanceToStaged` and *then* `promoteMemoryCandidateEntry` inside one `try`; the `catch` at `:136` records a skip reason but does not revert status. Preflight defers unverified candidates (`memory-inbox-service.ts:128-130`, reason `candidate_requires_verification`). The selector filters to `captured|candidate` only (`candidate-selector.ts:24`). Every fixture in `promote-gate.test.ts` is pre-set to `verified`, so the path is untested.
- **Fix.** Check verification (or run the full preflight) *before* `advanceToStaged`. On defer/deny, push to `result.skipped` with the reason and leave status untouched. Stronger option: wrap advance+promote in the transaction seam the inbox services already use and roll back on throw. Add a one-shot repair sweep that demotes already-stranded `staged_for_review` + unverified rows back to `candidate` (or lists them for the report).
- **Files.** `src/core/auto-promote/promote-gate.ts`, `src/core/services/memory-inbox-service.ts` (reuse preflight), `test/auto-promote/promote-gate.test.ts`.
- **Effort.** S–M.
- **Acceptance.** New test: an unverified candidate with a promote verdict ends still `candidate`, produces one `skipped{reason: candidate_requires_verification}`, and emits zero advance status-events. Existing verified-path tests unchanged.

#### GV-2 · Canonical handoff completion + real "promoted-but-unwritten" debt
- **Problem.** Promotion flips a candidate's status and records a handoff *intent* row, but the actual page write can fail silently afterward (patch-apply errors are swallowed into a skip reason). A candidate with a handoff row is then counted as fully resolved even though no retrievable markdown exists — exactly the "canonical debt" CLAUDE.md warns about, and it is invisible forever.
- **Evidence.** `promote-gate.ts:127` records the handoff, `:129` swallows `skipped_reason` from `canonicalizePromotedCandidate` *after* the handoff exists. `candidate-resolution-state-service.ts:68-69` treats `promoted_with_handoff` as resolved. `canonical_handoff_entries` has no completion column (`canonical-handoff-service.ts:51-61`). The report even claims to check "materialized page" but does not (`memory-review-report-service.ts:779-780`).
- **Fix.** Add `completed_at`, `completion_kind` (`patch_applied|page_written|manual`), `completion_ref` to `canonical_handoff_entries` (migration, all three engines). Stamp completion in the same flow as the successful write; leave it NULL on failure — that NULL *is* the debt record. `promoted_with_handoff` counts resolved only when `completed_at IS NOT NULL`; add a `promoted_handoff_incomplete` bucket surfaced in the report with a per-row next action (`apply_memory_patch_candidate <id>` / `put_page <slug>`). Optionally have dream maintenance retry failed applies.
- **Files.** `src/core/migrate.ts` (+ `schema.sql`), `canonical-handoff-service.ts`, `candidate-resolution-state-service.ts`, `memory-review-report-service.ts`, `src/core/auto-promote/promote-gate.ts`.
- **Effort.** M.
- **Acceptance.** Unit: handoff recorded + patch-apply throws → resolution state reports `promoted_handoff_incomplete`, report lists it. E2E: happy path stamps `completed_at`. Cross-engine migration coverage.

#### GV-3 · Actually generate and deliver the daily report
- **Problem.** The agent rules declare the daily memory report "the primary review surface," but nothing scheduled generates it. The 03:00 autopilot cron runs `autopilot run-once` (a dream cycle), whose `daily_report` phase only *counts* failed jobs and returns a recommendation string — it never renders or saves the report. If the owner never runs `mbrain report` by hand, no debt, contradiction, or dead-job streak ever surfaces.
- **Evidence.** Cron target is `autopilot run-once` (`setup-agent-autopilot.ts:114,274`). The dream `daily_report` phase is count-only (`dream-cycle-runner-service.ts:437-448`). The report builder has no non-CLI caller. Health beacon is read only by `doctor`; autopilot's only warning is a missing heartbeat.
- **Fix.** Inject the report builder into the dream `daily_report` phase (same DI pattern as the `lifecycleForgetting`/`autoPromote` deps), render via `memory-review-report-service`, and persist with `saveBrainReport` to `brain/reports/memory-report/`. Add delivery config `report.notify = { mode: 'auto'|'command'|'off' }`: `auto` = `osascript`/`notify-send` with a one-line summary (counts of candidates-awaiting-review, open contradictions, incomplete handoffs, dead jobs) + the report path; `command` pipes summary JSON to a user webhook command; silent no-op when unavailable. Add an alarm floor: warn (and beacon) when N≥2 consecutive nightly jobs die.
- **Files.** `dream-cycle-runner-service.ts`, `src/commands/report.ts` (reuse `saveBrainReport`), `src/core/config.ts`, a small `notify` helper, `autopilot-service.ts` / `health-beacon.ts`.
- **Effort.** M.
- **Acceptance.** Dream-cycle test with report deps + apply mode writes a report file and returns its path; notify-resolver unit test covers platform detection + command + off; doctor test flags 2 consecutive dead jobs. Ships with AR-2.

#### GV-4 · Expired write sessions fall back to a candidate
- **Problem.** When the router grants a canonical write, it creates a 15-minute write session holding the routed content. If the agent crashes or forgets to call `put_page`, that content — reserved for the *highest-authority* evidence class, direct user statements — lives only inside the expired session and is lost with no trace. The deferred-route branch gets a fallback captured candidate; the canonical branch gets nothing.
- **Evidence.** `operations-memory-writeback-router.ts:52` (`DEFAULT_MEMORY_WRITE_SESSION_TTL_MS = 15*60*1000`), `:330-364` (canonical branch creates a session, no fallback), `:370` (deferred branch calls `deferredRouteCandidateInput`). No sweep, report, or metric covers expired-unused sessions.
- **Fix.** In dream maintenance, sweep sessions past `expires_at` with no successful `put_page` bound to their `write_session_id`; convert each into a captured candidate via the existing `deferredRouteCandidateInput` builder, preserving `evidence_kind` (so `direct_user_statement` keeps its authority for later promotion) with `review_reason: 'expired_write_session_fallback'`. Mark the session swept (idempotent). List sessions swept last window in the report.
- **Files.** `operations-memory-writeback-router.ts`, the write-session store from `df6e48f`, the dream maintenance service, `memory-review-report-service.ts`.
- **Effort.** M.
- **Acceptance.** Unit: route canonical with `apply` → skip `put_page` → advance clock past TTL → sweep → a captured candidate exists with the routed content and reason `expired_write_session_fallback`; a second sweep is a no-op.

#### GV-5 · Timeouts + subprocess kill for maintenance/auto-promote
- **Problem.** No per-phase timeout or `AbortController` exists in the dream runner or maintenance runtime. `sweepTimedOutJobs` marks the leaked DB row `dead` but never cancels the running promise or its subprocess; idempotency dedup then returns `deduped` for every future cycle. A single hung phase — a stuck DB call, or a runner invoked through an injected `runCommand` that lacks the default timeout — silently blocks all future autopilot slots.
- **Evidence.** `dream-cycle-runner-service.ts:295`, `maintenance-runtime-service.ts:293-329`, `autopilot-service.ts:111-146`. Note the executor is *not* the gap: `auto-promote/cli-executor.ts` already self-limits its default spawn (120 s default `timeoutMs`, kill timer at `cli-executor.ts:23,63`) — but it exposes no abort seam, so the sweep cannot kill it early and injected `runCommand`s aren't covered. Runner-backed phases (5 claim-extraction, 9 contradiction, 15 auto-promote) already skip by default under `allow_local_runner:false` (`autopilot.ts:22-32`).
- **Fix.** Config `maintenance.phase_timeout_ms` (default 10 min). Wrap each phase in `Promise.race([phase, timeout])`; thread an `AbortSignal` into `RestrictedRunnerExecutor` so `proc.kill()` also fires on abort — the executor's own 120 s timer stays as the inner floor (register process handles in an in-process registry keyed by job id; `sweepTimedOutJobs` aborts still-registered handles). Record `timed_out: true` + duration per phase → surfaced by GV-3's report ("phase X timed out N nights running"), alongside standing skip reasons.
- **Files.** `dream-cycle-runner-service.ts`, `maintenance-runtime-service.ts`, `auto-promote/cli-executor.ts`, `src/core/config.ts`.
- **Effort.** M.
- **Acceptance.** Unit: a never-resolving fake phase with a 50 ms timeout → run completes, phase marked `timed_out`, next cycle is not `deduped`. cli-executor test: abort kills a spawned `sleep`.

### 3.2 Retrieval correctness

#### KR-1 · Unicode-aware search terms (SQLite)
- **Problem.** A pure-Korean query returns zero keyword results on the offline profile; a mixed query silently drops its Korean terms.
- **Evidence.** `sqlite-engine.ts:10147` — `term.split(/[^A-Za-z0-9]+/)` deletes all Hangul, so `prepareFtsQuery` receives an empty string and `searchKeyword` returns `[]`. The index itself is fine: `pages_fts` uses `tokenize='porter unicode61'` (`:230`), and the engine's own `"term"*` prefix syntax already matches particle-suffixed forms (verified: `"레벨리온"*` matches `레벨리온은`).
- **Fix.** Change the split to `/[^\p{L}\p{N}]+/u`. Keep ASCII alias expansion (`expandTechnicalAliases`) unchanged. Snippet term-matching uses the same helper and benefits for free.
- **Files.** `src/core/sqlite-engine.ts` (`extractSearchTerms`), `test/sqlite-engine.test.ts`, `test/retrieval-recall-harness.test.ts` (add Korean cases — they also seed EV-1 and KR-2).
- **Effort.** S.
- **Acceptance.** `extractSearchTerms('레벨리온 아키텍처')` returns both terms; a seeded Korean page is found by a pure-Korean query and by its particle-suffixed form; a mixed EN+KR query matches on both languages.

#### RQ-1 · `chunkSourceRankFactor` default branch
- **Problem.** The factor switch has no `default:`; a `chunk_source` outside the three known values returns `undefined`, which multiplies into `NaN`, which poisons the sort comparator and produces arbitrary result ordering. Unreachable via typed writers today, but the DB column is unconstrained `TEXT` and AL-2 introduces a fourth value (`'anchor'`).
- **Evidence.** `source-ranking.ts:60-68` (switch, no default), multiplication at `:93`. Prerequisite for AL-2.
- **Fix.** Add `default: return 1;`. Consider a CHECK constraint in the next migration.
- **Files.** `src/core/search/source-ranking.ts`, `test/` (a ranking unit test).
- **Effort.** S.
- **Acceptance.** A result with `chunk_source:'anchor'` ranks finitely and does not perturb the ordering of others.

### 3.3 Knowledge foundation

#### KM-1 · Two-zone timestamps → honest compile-debt
- **Problem.** The only compile-debt signal is `get_health().stale_pages` (`updated_at < max(timeline.created_at)`), and it is blinded: `put_page` bumps `updated_at` even when only the timeline zone grew, so the most common write path resets the signal to zero. It is also just an aggregate count — no op lists *which* pages, and one stale entry looks the same as twenty.
- **Evidence.** `pg-engine-base.ts:908-910` (stale_pages), `:300-301` (`put_page` sets `updated_at=now()`), `:706-713` (`add_timeline_entry` does not); SQLite parity `sqlite-engine.ts:1790`.
- **Fix.** Add `compiled_truth_changed_at` and `timeline_changed_at` to `pages` (migration, all engines; backfill both = `updated_at`). Every write path hash-compares each zone against the prior row (both are already parsed apart by `parsePage`) and stamps whichever changed. Compile debt per page = dated timeline lines newer than `compiled_truth_changed_at`, ranked by count × age. Expose a `list_compile_debt` op (tier: extended) and a `mbrain report --type compile-debt` section; repoint `stale_pages` at the new columns. This is KM-3's work queue and EV-2's groundedness proxy input.
- **Files.** `src/core/migrate.ts`, `schema.sql`, both engines' page-write methods, `operations.ts` (new op), `src/commands/report.ts`.
- **Effort.** M.
- **Acceptance.** A `put_page` that changes only the timeline bumps `timeline_changed_at` but not `compiled_truth_changed_at`; the page appears in the debt list with the correct entry count; cross-engine parity test for both columns.

#### KM-2 · Duplicate detection at write time and in sweep
- **Problem.** Candidate creation runs duplicate review; `put_page` does not. So direct page writes create duplicates freely — a page exists verbatim under `systems/mbrain-agentic-canonical-retrieval` and `office/systems/mbrain-agentic-canonical-retrieval` right now, both returned as separate retrieval candidates, splitting rank mass. Nothing sweeps for this (doctor and lint have no duplicate check).
- **Evidence.** `duplicate-memory-review-service.ts:77-131` (scoring exists); callers at `operations-memory-writeback-router.ts:371,403` and `memory-inbox-service.ts:138`; no caller in the `put_page` handler.

### 3.5 Security

Fundamentals are strong and must not regress (see §10.1). Five hardening items total — four MEDIUM (SC-1–SC-4) and one LOW (SC-5). SC-1–SC-3 land here; SC-4/SC-5 ride Wave 1 (§4.7):

#### SC-1 · Consent page shows who is asking
- **Problem.** The OAuth approval form lists requested scopes but never the client name or redirect URI. - **Evidence.** `renderApprovalForm` (`oauth.ts:422-454`) renders only the scope list.
- **Fix.** Render `client_name` and the `redirect_uri` origin (HTML-escaped, as scopes already are), with a distinct banner for a first-seen / previously-unapproved `client_id`.
- **Files.** `src/mcp/oauth.ts`.
- **Effort.** S.
- **Acceptance.** Unit on the rendered HTML contains the escaped client name and origin; an XSS attempt in `client_name` stays escaped.

#### SC-2 · `/mcp` connection reuse + rate limit
- **Problem.** Each `/mcp` request opens up to three fresh `postgres(url, {max:1})` clients — for token lookup, `last_used_at`, and request logging — triggered by *any* `Authorization: Bearer` header, valid or not. There is no rate limit on `/mcp` (only on `/oauth/*`). A bogus-token flood can exhaust `max_connections` on a connection-limited backend and take the owner's own brain offline.
- **Evidence.** `http-server.ts:396-440,442-523,649-675`. The live engine pool is available (`PostgresEngine.sql`, `postgres-engine.ts:34`); the Edge function already reuses a module-level connection (`supabase/functions/mbrain-mcp/index.ts:58-93`).
- **Fix.** Reuse `(await enginePromise).sql` (or one shared `max:3` pool created at startup) for the three helpers; add a per-IP token bucket on `/mcp` mirroring the existing OAuth limiter (generous — DoS protection, not throttling).
- **Files.** `src/mcp/http-server.ts`.
- **Effort.** M.
- **Acceptance.** N requests do not create N postgres clients (spy the factory); `/mcp` returns 429 after the bucket empties; the OAuth-flow smoke is unchanged.

#### SC-3 · Approval-token entropy floor
- **Problem.** Docs say "choose a long random token," but code accepts a 4-digit PIN, which the shared 10/min authorize limiter cannot protect.
- **Evidence.** `oauth.ts:552-559`, startup validation `serve.ts:73-86`.
- **Fix.** Reject `MBRAIN_OAUTH_APPROVAL_TOKEN` shorter than 16 chars at startup, matching the existing signing-secret fail-closed check.
- **Files.** `src/commands/serve.ts`.
- **Effort.** S.
- **Acceptance.** An 8-char token refuses to start; a 16-char token passes.

## 6. Wave 3 — Measured retrieval changes **[GATE]**

Nothing here flips a default without an EV-1 non-regression artifact on the DG-1 enabled-flags configuration.

- **RQ-7 · Flip `retrieval_governed_probe_hybrid` default-on · S · DG-1.** Wired since v0.14.3 (`governed-probe.ts:22,40`), degrades to keyword-only without a provider, but still default-off (`config.ts:134-136`) — so a rule-following `retrieve_context` gets strictly worse recall than the lower-authority `query` op. **Enable it now for this install via `setup-agent`** (Korean semantic recall shouldn't wait); the *global* default flips on the EV-1 artifact. Pair with RQ-3 for latency.
- **RQ-8 · Cross-engine score normalization · M.** `scoreResolvedCandidate` base = max chunk score × 100, but the chunk score is `ts_rank` (~0.01–1) on PG and `|bm25|` (~1–15) on SQLite, so identical flat bonuses (+18/+16..64/+12) dominate on PG and are noise on SQLite — Invariant 8 ranking parity is broken (`retrieve-context-service.ts:862-889`, `sqlite-engine.ts:10109`). Normalize the base (rank-based or pool min-max) before adding bonuses; assert cross-engine ordering in EV-1 fixtures.
- **RQ-9 · Collapse RRF vector-variant lists · S.** Each expansion variant's vector list is a separate RRF list, so a chunk in all three earns up to 3×1/60 vs the single keyword list's 1/60; K is also inconsistent (`1/(60+rank)` vs `1/(61+rank)` at `retrieve-context-service.ts:722`). Take the max across vector variant lists; align K.
- **RS-7 · [FLAG] Local reranker · L · alternative to RF-1.** A cross-encoder (qwen3-reranker via the llama.cpp HTTP pattern) evaluated at both seams — the post-RRF chunk seam (`hybrid.ts:106-110`, between fusion and dedup) and the candidate seam (a `RetrieveContextDependencies` override) — off by default, deterministic order when absent. Matters because `required_reads` caps at 3, so reranking decides what the agent actually reads. Choose RS-7 *or* RF-1 on the EV-1 judge lane — do not stack.
- **RS-8 · Port SQLite's two-stage shortlist to Postgres · M.** SQLite prefilters by `page_embedding` centroid (8× limit, min 25, omitted-top rescue — `vector-prefilter.ts:13-35`), while PG does a flat chunk scan and never reads `page_embedding`. Port the semantics into a CTE in `PgEngineBase.searchVector` — Invariant 8 parity *improves* and `page_embedding` stops being maintained-but-unread on the target runtime.
- **RW-5 · Map reads compete for `required_reads` · M.** Context-map `recommended_reads` are quarantined in `orientation` and can never change what gets read (`retrieve-context-service.ts:1510-1528`; `broad-synthesis-route-service.ts:43-72`). Let them compete for `requiredReadLimit` slots (or boost candidates sharing a map community with seed hits) on synthesis-shaped queries.
- **RW-6 · Auto-route · M.** `retrieve_context` classifies the query's scenario and returns `route:null` regardless (`retrieve-context-service.ts:172-332`); the route/planner ops must be called separately and applied by hand. Auto-apply the obvious mappings (personal scenarios → personal routes; precision vs synthesis → candidate/read limits) with an `auto_route:false` opt-out.

**The Memora index lanes — build only if EV-1 shows a residual gap (DG-8).** RW-1 + RS-2 + RS-6 together deliver most of the abstraction/cue-anchor benefit deterministically and for free. Run the A/B first.
- **AL-1 · [FLAG] Abstraction-handle lane (Memora #1) · L.** Additive `abstraction_handles(page_id, handle_text, embedding, …)`, handle built deterministically (title + slug + aliases + first compiled-truth sentence), added as a third RRF leg weighted above body chunks, resolving to the `compiled_truth` chunk so evidence stays canonical. *Risk:* near-zero marginal value once RS-2 ships (both give page identity in vector space).
- **AL-2 · [FLAG] Cue-anchor lane (Memora #2) · L · needs RQ-1.** `chunk_source='anchor'` rows in `content_chunks` (same `page_id` FK), generated deterministically (title/slug/aliases) with optional LLM enrichment in the `enrich` skill (1–3 distinct anchors, never generic, never a title repeat; fail-closed to the deterministic set); rank factor ~1.06. **Must be excluded from every `read_context` selector** (constraint 2). Overlaps RW-1/RS-6 — do it with AL-1 or not at all.
- **RF-1 · [FLAG] Relevance filter (Memora #7) · M · alternative to RS-7.** A 1–3 relevance judge after `rankResolvedCandidates` (drop score<2), provider resolved via a `resolveRelevanceFilterPolicy` seam mirroring `resolveQueryRewritePolicy` (identity offline), config `retrieval.relevance_filter.enabled` default off, applying Memora's fail-open guards verbatim (mismatched count or any throw → return all unchanged), on broad/synthesis routes only, never pruning a `required_read` or frontier candidate. Value is token savings + tighter reads, not a correctness leap — which is why it competes with RS-7.

- **GV-10 · Post-promotion refutation · M.** `verifyMemoryCandidateEntry` rejects terminal statuses (`memory-inbox-service.ts:261-266`), so a claim found false after promotion has no correction path short of full supersession + manual page edit. Allow refutation of promoted candidates that flags the handoff/page line for review.

### 7.2 Architecture (fill-in work; no user-visible risk)
- **AR-3 · Split `operations.ts` · S/M.** 8,287 lines / 189 ops (54% of all 349); the `// --- Operational Memory ---` section (`:5418`→~7801, ~2,383 lines) extracts to `operations-tasks.ts` / `operations-note-manifest.ts` following 12 sibling precedents. ## 9. Non-goals (and why)

- **SQLite/Postgres engine-code unification** — the dialect divergence is real, not copy-paste (verified); parity-matrix tests (AR-8) are the guardrail, not shared SQL.
- **Korean morphological analysis in-engine** (mecab/kiwi) — non-portable; prefix legs (KR-2) + the multilingual vector path are the answer.
- **Resurrecting the retired assertion apply path** — ADR 2026-06-22 stands; GV-2 closes the debt on the live ladder instead.
- **Autonomous compiled-truth rewriting** — KM-3 routes through patch candidates + review by design.
- **A bi-temporal knowledge-graph store** (Zep/Graphiti-style) — RS-4 + KM-4 + `page_versions` deliver the queryable slice; revisit only if temporal queries dominate EV-1 failures.
- **Late-interaction retrieval (ColBERT) / a new vector DB** — unjustified at personal-brain scale before RS-2/RS-7 results exist.
- **Memora's non-portable pieces** — the GRPO/RL policy (research, not product; its one nugget `J=G−R−C` is EV-2); the in-engine LLM retrieval loop (the caller *is* the loop; adds latency, breaks offline parity); delete-and-rewrite JSON history (regresses the governed, hash-gated model); dangling-pointer graphs (mbrain's FKs give integrity for free); Memora's exact tuning constants (adopt as tunable defaults only). Its low-value MONITOR ideas (deterministic `suggested_followups`, wiring the dead semantic/LLM chunkers, factual→episodic FK provenance) are deferred pending a concrete EV-1 trigger.
- **Multi-user auth/RLS; cloud-OAuth capture connectors** — owned elsewhere / deferred (local-file connectors first).
- **Re-speccing the 2026-06-24 strategic spec's open items** — B2, C2, D1/D4/D5, F3/F4 keep their ownership there.
