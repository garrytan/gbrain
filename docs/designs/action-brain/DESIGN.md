## Problem Statement

Abhi is a commodity trader and mining operator running multiple businesses (Denare, DM Tin, KTM Tanzania) with partners (Sagar, Nichol), site managers (Joe, Mukesh), and personal operations. He operates in "mental mode" — keeping everything in his head, managing through WhatsApp, with scattered tools and no integrated system. He previously hired a PA but it didn't work out because the PA couldn't self-serve context from message threads. He has ADHD, making mental-mode operation actively dangerous for execution quality.

The core problem is not "too many messages." It is that commitments, follow-ups, and obligations are scattered across WhatsApp threads, and the founder manually stitches them into an operating picture every day. Dropped loops in mining/commodity operations have real financial and reputational consequences.

## Recommended Approach

**Approach B-Prime: GBrain-Native Extension.**

Build the Action Brain as an extension alongside GBrain, sharing the same database and patterns. Prove commitment extraction quality first. Graduate to full stack integration (A or C) only after the morning brief is trustworthy.

Key architecture decisions:
- Same PGLite database instance, separate tables for action items
- Shared entity references (people/company/project pages in GBrain)
- Unified query surface (CLI + MCP)
- Reuse GBrain recipe patterns: deterministic collector, idempotent ingestion, entity linking, scheduled enrichment
- NEW subsystem: item reconciliation/state-transition engine

## Data Model

### GBrain Pages (knowledge — existing)
People, companies, projects, concepts. Provides context for obligations.

### Action Items (obligations — new)
```
action_item:
  id: auto
  title: text
  type: commitment | follow_up | decision | question | delegation
  status: open | waiting_on | in_progress | stale | resolved | dropped
  owner: text (who must act)
  waiting_on: text (who we're waiting for)
  due_at: timestamp
  stale_after_hours: int (default 48)
  priority_score: float (auto-calculated)
  confidence: float (extraction confidence)
  source_message_id: text
  source_thread: text
  source_contact: text
  linked_entity_slugs: text[] (GBrain page slugs)
  created_at: timestamp
  updated_at: timestamp
  resolved_at: timestamp

action_history:
  item_id: FK
  event_type: created | status_change | reminded | escalated | resolved | dropped
  timestamp: timestamp
  actor: text (system or user)
  metadata: jsonb
```

### Links Between Them
- Every action item links to relevant GBrain entity pages
- GBrain person pages show their open obligations when queried
- Unified query: "What's going on with Sagar?" returns both knowledge AND obligations

## Pipeline Architecture

```
WhatsApp Messages (via Pulse, 15-min polling)
  |
  v
Deterministic Collector (code, not LLM)
  - Dedup by message ID
  - Filter noise
  - Group by contact/thread
  |
  v
Commitment Extractor (LLM, one call per batch)
  - Extract: {who, owes_what, to_whom, by_when, confidence}
  - Classify: commitment / follow_up / decision / question
  |
  v
Item Reconciler (the hard part)
  - New obligation? → create item
  - Update existing? → change status/due/owner
  - Evidence of resolution? → mark resolved
  - Duplicate? → skip
  - Idempotent by source_message_id + content hash
  |
  v
GBrain Entity Linker
  - Match mentioned people/companies to GBrain pages
  - Attach context to each item
  |
  v
Priority Scorer
  - Staleness, overdue, relationship weight, monetary significance
  - Reuse Mission Control's 9-component scoring model
  |
  v
Brief Generator (scheduled)
  - Morning: top 5 priorities + stale items + person-level summary
  - Evening: unresolved items from today
  - Delivery: Telegram (MVP), WhatsApp later
```

## MVP 0.1 Scope

Ship: WhatsApp action intelligence
- Ingest WhatsApp messages via Pulse
- Extract commitments with LLM (one call per message batch)
- Classify: waiting_on, owed_by_me, urgent_today, stale
- Store as action items in PGLite
- Link to GBrain person/project pages
- Morning priority brief (delivered via WhatsApp/Telegram)
- Person-level context from GBrain
- Traceability: every item links to source message

Skip in 0.1:
- Suggested responses
- Calendar integration
- Email ingestion
- Multi-user
- UI beyond the brief itself

## MVP 0.2 Scope

Ship: Suggested responses
- Draft replies for high-priority items
- Context-aware (knows the person, history, stakes from GBrain)
- Confirm-before-send (Abhi approves, system sends)

## Failure Modes & Recovery

| Stage | Failure | Behavior |
|-------|---------|----------|
| Pulse collector | WhatsApp polling fails | Brief reports "ingestion stale since {time}". No false confidence. |
| LLM extraction | API timeout or error | Raw messages queued for next batch. Brief notes "N messages pending extraction." |
| Item reconciler | Ambiguous match | Default to creating new item. False duplicates are less dangerous than missed items. |
| GBrain unavailable | Entity linking fails | Items created without entity links. Brief omits person context. System runs degraded, not broken. |
| PGLite write failure | DB contention or corruption | Retry once. On persistent failure, log error and alert. |
| Brief delivery | Telegram/WhatsApp send fails | Retry 3x with backoff. Fall back to alternate channel if primary fails. |

Action Brain starts and runs without GBrain. If GBrain is available, entity linking and person-level context enrich the brief. If unavailable, items are created without entity links and the brief omits person context.

## User Correction Interface (MVP 0.1)

Abhi marks false positives/negatives via CLI:
- `action mark-fp <id>` — mark extraction as false positive (wrong commitment detected)
- `action mark-fn <message-id>` — flag a message that should have produced an item but didn't
- `action resolve <id>` — manually resolve an item
- `action snooze <id> <hours>` — push staleness window forward

Corrections feed back into extraction accuracy metrics. Target: 90%+ precision on real data.

## Reconciliation Strategy (MVP)

MVP reconciliation uses simple heuristics, not fuzzy LLM matching:
1. Exact source_contact + same topic keywords within 48h window = candidate update
2. If candidate found, compare extracted fields. If status-change-like ("done", "sent", "confirmed"), update status
3. Otherwise, create new item
4. Defer fuzzy reconciliation ("I sent Joe the thing" = resolves item #47) to MVP 0.2

## Stale Transition Rules

When an item exceeds `stale_after_hours` (default 48h):
- Status changes to `stale`
- Item appears in morning brief under "Gone Stale" section
- History entry logged with event_type `escalated`
- No automatic nudge in MVP (nudge is MVP 0.2)

## Data Retention

- Resolved/dropped items older than 30 days move to `action_items_archive` table
- Archive is queryable but excluded from brief generation and priority scoring
- No hard deletes in MVP

## Classification Taxonomy

Two separate taxonomies, not one:
- **Type** (what it is): commitment | follow_up | decision | question | delegation — set at extraction
- **Status** (lifecycle): open | waiting_on | in_progress | stale | resolved | dropped — set by reconciler/scorer
- **Brief buckets** (user-facing view): waiting_on_others | owed_by_me | urgent_today | gone_stale — computed from type + status + due_at + staleness

## Priority Scoring (MVP)

Simple weighted sum, tuned later:
- staleness_score (weight 30): hours since last activity / stale_after_hours
- overdue_score (weight 40): max(0, hours_past_due / 24)
- relationship_weight (weight 15): based on contact frequency and GBrain relationship data
- monetary_flag (weight 15): binary, set if financial terms detected in source message

Total = sum of weighted components. Top N by score appear in morning brief.

## Latency Constraint

Pulse polls WhatsApp every 15 minutes. This means commitments may be up to 15 minutes stale on ingestion. Acceptable for MVP — morning briefs are generated from accumulated state, not real-time. For time-sensitive items ("I'll send in 10 minutes"), the system catches them on the next poll and tracks from there.

## PGLite Sharing Model

MVP: GBrain and Action Brain run in the same process (shared PGLite instance). Separate tables, shared connection. No contention issue for single-user local use. If later separated into independent processes, will need connection arbitration or separate PGLite instances.

## Brief Delivery

MVP default: Telegram (bot sending is trivial, already wired via OpenClaw). WhatsApp delivery deferred — Pulse is read-only (monitor), not a sender. WhatsApp Business API integration is a later phase.

## Open Questions

1. Extraction accuracy: can the LLM reliably extract commitments from real WhatsApp conversations at 90%+ accuracy? This is the gating question.
2. Reconciliation complexity: how hard is it to decide "this message updates existing item X" vs "this is a new obligation"? May need iterative refinement.
3. WhatsApp delivery: can we reliably send the morning brief back via WhatsApp, or is Telegram the safer delivery channel?
4. Mission Control migration: 405 existing tasks in Mission Control. When and how do they migrate into Action Brain?
5. Scoring model: MVP uses simple 4-component weighted sum (defined above). May evolve toward Mission Control's 9-component model if complexity is justified by usage.

## Success Criteria

1. Abhi uses the morning brief daily for at least 2 weeks without reverting to mental mode
2. Commitment extraction accuracy >= 90% on real WhatsApp data (measured by Abhi marking false positives/negatives)
3. At least one "this saved me from dropping something important" moment
4. Sagar or Nichol, when shown the brief, says "I want this"

## The Assignment

Before building anything: run the extraction quality test.

Take 50 real WhatsApp messages from the last week. Feed them to an LLM with a commitment extraction prompt. Manually grade every extracted item: correct, false positive, or missed. Calculate precision and recall.

If precision >= 90% and recall >= 80%, the product thesis holds. Build it.
If not, iterate on the extraction prompt until it does. Do not build infrastructure until extraction quality is proven.

This test takes 2 hours, not 2 weeks. Do it before writing a single line of product code.
