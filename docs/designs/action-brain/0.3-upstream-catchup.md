## Why this order is locked

1. `#231` is the upstream ancestry base (`c22ca84`) and introduces typed frontmatter edges/schema updates needed before later runtime work in this wave.
2. `#210` (`c89aa90`) follows `#231` in ancestry and introduces the Knowledge Runtime substrate (Resolver SDK, BrainWriter path, integrity command, Budget/Completeness plumbing) used by later integrations.
3. `#217` (`5fd9cd2`) follows `#210` in ancestry and extends minions with shell handlers plus abort-path hardening; landing it before integration avoids later queue-runtime drift.
4. Resolver SDK adoption in `action_brief` (Child 4) is first integration step because later linking/budget behavior depends on stable context sourcing.
5. `action_drafts` typed-edge wiring (Child 5) depends on both typed-edge schema and resolver-backed context behavior.
6. Budget-layer adoption (Child 6) follows context/linking updates so budget controls apply to the final integrated draft path.
7. Security-invariant audit (Child 7) runs last so v0.13.0 guarantees are validated on the final integrated implementation, not a partial state.

## Risks and containment

- Conflict risk: this fork already diverged in commands/core. Containment: keep cherry-picks narrow, document every manual conflict resolution.
- Integration drift risk: upstream absorption can land while Action Brain keeps legacy internal paths. Containment: lock Child 4-7 as required scope before close.

- Security regression risk: context/linking/budget rewires can bypass prior hardening. Containment: Child 7 audit + explicit regression tests before close.
