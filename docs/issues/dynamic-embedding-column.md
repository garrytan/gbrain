# Dynamic Embedding Column Selection for Search

## Summary

GBrain's search pipeline hardcodes `embedding` (OpenAI text-embedding-3-large, 1536d) as the vector column. Users with multiple embedding providers (Voyage, ZeroEntropy, etc.) have fully backfilled alternative columns (`embedding_voyage`, `embedding_zeroentropy`) with valid HNSW indexes, but search cannot route queries through them. This change adds a `search_embedding_column` config key and extends the `embeddingColumn` SearchOpts type to support any declared embedding column — enabling A/B benchmarking, provider failover, and per-query column routing.

## Problem

**What the user sees:** Search always uses OpenAI embeddings regardless of which providers are configured and backfilled. Users cannot compare search quality across providers or switch to a better/cheaper provider without code changes.

**What the system does:** `postgres-engine.ts:searchVector()` (line ~1133) resolves the column as:
```typescript
const col = opts?.embeddingColumn === 'embedding_image' ? 'embedding_image' : 'embedding';
```
This is a binary switch: `embedding` or `embedding_image`. No third option.

The `SearchOpts.embeddingColumn` type in `types.ts` (line ~491) is:
```typescript
embeddingColumn?: 'embedding' | 'embedding_image';
```

The query embedding is produced by `embedQuery()` in `embedding.ts`, which always calls the configured `embedding_model` (e.g., `openai:text-embedding-3-large`). Even if the column were switchable, the query vector would be in the wrong embedding space for non-OpenAI columns.

**What it should do:**
1. Config key `search_embedding_column` selects the default column for search (default: `'embedding'` for backward compat)
2. The column name maps to an embedding provider, so query embeddings are generated by the matching provider
3. Per-query override via `SearchOpts.embeddingColumn` for A/B testing
4. Fallback: if the selected column has NULL embeddings for a chunk, those chunks are excluded (the `WHERE cc.${col} IS NOT NULL` clause already handles this)

## Evidence

### Three embedding columns exist with full coverage

```sql
-- Production state (May 2026, ~375K chunks)
SELECT
  COUNT(*) FILTER (WHERE embedding IS NOT NULL) as openai,        -- 371,887
  COUNT(*) FILTER (WHERE embedding_voyage IS NOT NULL) as voyage,  -- 247,653
  COUNT(*) FILTER (WHERE embedding_zeroentropy IS NOT NULL) as ze  -- 374,486
FROM content_chunks;
```

### HNSW indexes exist for all three

```
✅ idx_chunks_embedding              2308 MB  (OpenAI 1536d, vector)
✅ idx_chunks_embedding_voyage       3542 MB  (Voyage 1024d, vector)
🔄 idx_chunks_ze_halfvec_hnsw       ~850 MB  (ZE 2560d, halfvec — building)
```

### Current search only uses OpenAI

`postgres-engine.ts` line ~1133:
```typescript
const col = opts?.embeddingColumn === 'embedding_image' ? 'embedding_image' : 'embedding';
```

### Query embedding always uses the configured model

`embedding.ts` line 34-36:
```typescript
export async function embedQuery(text: string): Promise<Float32Array> {
  return gatewayEmbedQuery(text);
}
```

`gateway.ts embed()` resolves the provider from `getEmbeddingModel()` which reads `cfg.embedding_model` — a single global setting.

## Root Cause Analysis

The embedding column was designed as a binary text/image switch in v0.27.1. The type system enforces this:

- **`types.ts:491`** — `embeddingColumn` union type is `'embedding' | 'embedding_image'`
- **`postgres-engine.ts:~1133`** — Column resolution is a ternary, not a lookup
- **`embedding.ts:34-36`** — `embedQuery()` has no provider parameter; it always uses the global config
- **`gateway.ts:embed()`** — Provider resolution uses `getEmbeddingModel()` with no override mechanism

There is no column→provider mapping anywhere. The system assumes one embedding model produces all vectors in the `embedding` column.

## Proposed Fix

### 1. Column registry (config layer)

Add an `embedding_columns` config map that declares available columns and their providers:

```yaml
# .gbrain.yml or config set
embedding_columns:
  embedding:                          # Default, always present
    provider: openai:text-embedding-3-large
    dimensions: 1536
    type: vector
  embedding_voyage:
    provider: voyage:voyage-3-large
    dimensions: 1024
    type: vector
  embedding_ze_half:                  # Use the halfvec column, not the raw 2560d one
    provider: zeroentropyai:zembed-1
    dimensions: 2560
    type: halfvec

# Which column to use for search by default
search_embedding_column: embedding    # Default: backward compat
```

The registry maps column name → provider + dimensions + pgvector type. This is the single source of truth.

### 2. Widen the type (types.ts)

```typescript
// Before
embeddingColumn?: 'embedding' | 'embedding_image';

// After
embeddingColumn?: string;  // Column name from embedding_columns registry, or 'embedding_image' for multimodal
```

### 3. Column resolution (postgres-engine.ts)

Replace the ternary with a registry lookup:

```typescript
// Before
const col = opts?.embeddingColumn === 'embedding_image' ? 'embedding_image' : 'embedding';

// After
const col = resolveEmbeddingColumn(opts?.embeddingColumn);
// resolveEmbeddingColumn: opts value > config search_embedding_column > 'embedding'
```

The `resolveEmbeddingColumn()` function:
1. Validates the column exists in the registry
2. Returns the column name for SQL interpolation
3. Determines the correct operator class (`vector_cosine_ops` vs `halfvec_cosine_ops`)

### 4. Query embedding routing (embedding.ts / gateway.ts)

`embedQuery()` needs a provider override:

```typescript
// Before
export async function embedQuery(text: string): Promise<Float32Array> {
  return gatewayEmbedQuery(text);
}

// After
export async function embedQuery(text: string, provider?: string): Promise<Float32Array> {
  return gatewayEmbedQuery(text, { provider });
}
```

And `gateway.ts embed()` needs to accept an optional provider override that bypasses `getEmbeddingModel()`.

### 5. Wire it through hybrid.ts

`hybridSearch()` resolves the active column from config/opts, then:
1. Passes the column to `searchVector()` (already does this via `embeddingColumn`)
2. Looks up the provider from the registry
3. Passes the provider to `embedQuery()` so the query vector matches the column's embedding space

### 6. Cast handling for halfvec

When the column type is `halfvec`, the query vector needs casting:

```sql
-- vector column
ORDER BY cc.embedding <=> $1::vector

-- halfvec column
ORDER BY cc.embedding_ze_half <=> $1::halfvec(2560)
```

The `resolveEmbeddingColumn()` function returns the cast type alongside the column name.

### 7. Search mode integration (optional, can defer)

Add `search_embedding_column` to the mode knob set so different modes can use different providers:

```typescript
// tokenmax could use ZE (better recall) while conservative uses OpenAI (proven)
tokenmax: { search_embedding_column: 'embedding_ze_half', ... }
conservative: { search_embedding_column: 'embedding', ... }
```

This is optional for v1 — global config is sufficient to start.

## Test Guidance

### Red tests (should fail before fix, pass after)

1. **Config parsing:** Set `search_embedding_column: 'embedding_voyage'` in config. Verify `resolveEmbeddingColumn()` returns `'embedding_voyage'` (not `'embedding'`).

2. **Type widening:** Pass `embeddingColumn: 'embedding_voyage'` in SearchOpts. Should not throw a type error.

3. **Provider routing:** Call `embedQuery('test', 'voyage:voyage-3-large')`. Verify the returned vector has 1024 dimensions (Voyage), not 1536 (OpenAI).

4. **SQL generation:** With `embeddingColumn: 'embedding_ze_half'`, verify the generated SQL uses `cc.embedding_ze_half <=> $1::halfvec(2560)`, not `cc.embedding <=> $1::vector`.

5. **Fallback:** With no config override, verify search still uses `embedding` column (backward compat).

6. **Invalid column:** Set `search_embedding_column: 'nonexistent'`. Should throw a config validation error, not silently fall back.

### Edge cases

- Column exists in config but no HNSW index → search works but is slow (sequential scan). Doctor should warn.
- Column exists but has NULL embeddings for some chunks → those chunks excluded from results (existing WHERE clause handles this, but verify).
- Halfvec column with vector query cast → verify pgvector accepts the cast and uses the correct index.
- Multimodal column (`embedding_image`) should still work alongside the new system.

### Regression tests

- All existing search tests pass unchanged (default column is still `embedding`).
- `embedding_image` multimodal routing still works.
- Cache key includes the column name (different columns = different cache entries).

## Reproduction Steps

```bash
# Current: search always uses OpenAI embeddings

bun run src/cli.ts search "test query" --debug
# Output shows: column=embedding, provider=openai:text-embedding-3-large

# After fix: should be configurable
bun run src/cli.ts config set search_embedding_column embedding_voyage
bun run src/cli.ts search "test query" --debug

```

## Related Context

- PR #1008 added ZeroEntropy zembed-1 + zerank-2 support (v0.35.0.0)
- v0.27.1 introduced the `embeddingColumn` dual-column system for text/image
- The `embedding_columns` registry pattern mirrors how `models.tier` already maps use cases to providers
- Reranker (zerank-2) is independent of the embedding column — it works on text, not vectors. The fix only affects the vector search stage.
- Query cache (`query-cache.ts`) keys by `(source_id, query_text)` + `knobs_hash`. The `knobs_hash` must include the column name so different columns don't share cache entries.
