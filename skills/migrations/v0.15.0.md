# MBrain 0.15.0 Migration

MBrain 0.15.0 ships schema v64 and makes the July 2026 governance, retrieval,
review UI, watched-question, and compile-debt wave visible to existing installs.

## Required

1. Pull or install MBrain 0.15.0.
2. Apply pending schema migrations to the already configured brain.

Local owner-managed installs (SQLite, PGLite, and loopback Postgres) also
auto-apply these migrations the next time `mbrain` connects. To run the
migration explicitly, use the same target that is already stored in
`~/.mbrain/config.json`:

```bash
mbrain init --url "$MBRAIN_DATABASE_URL" --non-interactive
# or, for legacy local profiles:
mbrain init --local
mbrain init --pglite
```

Do not use bare `mbrain init` as a remote migration command; it may enter the
interactive setup wizard instead of targeting the existing database. For remote
Postgres, pass the existing configured DSN via `--url` or
`MBRAIN_DATABASE_URL`/`DATABASE_URL`.

Fresh 0.14.3 installs usually need v61-v64. Older live installs may need v59-v64;
the owner's July 4 live database was observed at schema v58.

3. Verify schema freshness:

```bash
mbrain call get_health '{}'
mbrain doctor --agent --explain --json
```

Expected: `schema_up_to_date` is true and the doctor `schema_version` check is
ok for schema v64.

4. Re-register agent/autopilot surfaces so scheduled healing and review reports
use the 0.15.0 command path:

```bash
mbrain setup-agent --apply
```

## Review The New Flags

Check or set these config values intentionally after migration:

- `retrieval.governed_probe_hybrid`
- `retrieval.contextual_chunk_embeddings`
- `retrieval.usage_aware_ranking`
- `retrieval_eval.answer_grounding`
- `maintenance.governed_recompile_enabled`
- `retrieval.source_rank_rules`

The governed hybrid probe remains enabled by default in 0.15.0 for parity with
the shipped wave, but ranking changes after this release should be gated by a
real retrieval eval baseline.

## Review UI

`mbrain review` now prints a per-session token. Open the printed URL or pass the
token as `Authorization: Bearer <token>` when calling review API routes. Binding
to a non-loopback host requires an explicit `--i-know` acknowledgement.
