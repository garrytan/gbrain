# v0.11.0 Migration: Postgres Runtime Governance

This release makes Postgres + pgvector the target runtime for new installs and
adds the governed memory loop around it: source policy, raw ingest provenance,
assertion extraction, canonical-write guards, lifecycle forgetting, restricted
runners, review reports, and replay evaluation.

Existing SQLite or PGLite installs are not forcibly migrated. Treat them as
legacy/offline compatibility profiles unless the user explicitly asks to move to
the Postgres target runtime.

## What's New

### Governed Postgres Runtime

MBrain now has a durable runtime for:

- source registry policy and raw access ledgers
- claim extraction and assertion resolution
- governed canonical writes with target snapshot preconditions
- maintenance jobs, autopilot, and dream-cycle readiness reporting
- lifecycle forgetting and tombstone audit trails
- restricted runner execution with scoped raw access
- system-of-record projection reconciliation
- daily review reports and deterministic replay checks

### Honest Legacy Compatibility

SQLite and PGLite remain available for explicit local/offline compatibility, but
new Postgres-only runtime surfaces should fail with clear guidance instead of
pretending every backend supports every feature.

## Agent Instructions

When presenting this update, say something like:

"v0.11.0 gives your brain a governed Postgres runtime. New installs use
Postgres + pgvector as the target path, while existing SQLite/offline installs
keep working as legacy compatibility profiles. I can inspect your current setup
and help you decide whether to stay local/offline or move to Postgres."

## Step 1: Inspect the Current Install

Run:

```bash
mbrain doctor --json
```

Check the active profile:

- `engine: "postgres"` means the user is already on the target runtime.
- `engine: "sqlite"` or `engine: "pglite"` means the user is on the
  legacy/offline compatibility path.

Do not switch engines without explicit user approval.

## Step 2: Refresh Agent Rules

Refresh injected agent rules so the user's clients learn the Postgres target
runtime and legacy/offline distinction:

```bash
mbrain setup-agent --skip-mcp
```

If the user wants MCP registration refreshed too, run the normal setup-agent
flow instead:

```bash
mbrain setup-agent
```

## Step 3: If the User Is Already on Postgres

If `mbrain doctor --json` reports `engine: "postgres"`, keep the existing
profile and apply the current schema migrations before enabling remote OAuth:

```bash
mbrain init --url '<existing database_url from ~/.mbrain/config.json>' --non-interactive
```

Then verify:

```bash
mbrain doctor --json
```

`mbrain serve --http --oauth` also prepares the schema before opening OAuth
routes, but running `init` explicitly gives the user a clearer post-upgrade
checkpoint and catches connection or pgvector problems before a remote client
tries to register.

## Step 4: If the User Wants the Postgres Target Runtime

Only continue after the user confirms they want to move from legacy local/offline
mode to Postgres.

Back up the current config:

```bash
mkdir -p ~/.mbrain/backups
cp ~/.mbrain/config.json ~/.mbrain/backups/config.pre-v0.11.0.json
```

Create or select a Postgres database:

```bash
createdb mbrain 2>/dev/null || true
```

Initialize the target profile:

```bash
mbrain init --profile homebrew-postgres
```

Import the user's canonical Markdown brain repo:

```bash
mbrain import ~/git/brain
```

If their repo lives somewhere else, replace `~/git/brain` with the actual
Markdown source path.

Verify:

```bash
mbrain doctor --json
mbrain stats
mbrain search "a term that should exist in my notes"
```

Important: `mbrain migrate --to postgres` is a Markdown-first migration guide,
not an automatic legacy DB page copier. Legacy DB-only state must be reviewed
before it becomes canonical memory.

## Step 5: If the User Stays on Legacy Local/Offline

No migration is required. Confirm the profile is intentional:

```bash
mbrain doctor --json
```

Explain that SQLite/PGLite remains useful for offline compatibility, but new
Postgres runtime features may report unsupported capability guidance.

Record completion:

```bash
mkdir -p ~/.mbrain/migrations
```
